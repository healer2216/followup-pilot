## 12. RAG Pipeline：循证检索增强生成

### 12.1 设计理念

不再直接调用 KnowS API 就用返回结果，而是构建完整的 **检索-增强-生成** 管道：

```
Query 构造 → KnowS 多源检索 → 结果向量化 → 相关性重排序 → Top-K 上下文注入 → LLM 生成
```

### 12.2 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                        RAG Pipeline                              │
│                                                                  │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────────────┐ │
│  │ QueryBuilder │──▶│ KnowSRetriever│──▶│ EmbeddingIndexer     │ │
│  │ 查询构造器    │   │ 多源检索器    │   │ 向量化 + 内存索引     │ │
│  └──────────────┘   └──────────────┘   └──────────┬───────────┘ │
│                                                     │            │
│                                          ┌──────────▼───────────┐│
│                                          │ Reranker             ││
│                                          │ 交叉编码器重排序      ││
│                                          └──────────┬───────────┘│
│                                                     │            │
│                                          ┌──────────▼───────────┐│
│                                          │ ContextInjector      ││
│                                          │ 上下文组装 + Token控制││
│                                          └──────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
```

### 12.3 核心组件

#### QueryBuilder（查询构造器）
- 基于 **PICO 框架**（Population/Intervention/Comparison/Outcome）构造查询
- 每种药物单独查说明书 + 多药联合用药查询
- 老年/特殊患者自动生成定制查询
- 每个查询标注优先级（指南 > 说明书 > 论文）

#### EmbeddingIndexer（向量化索引器）
- 将证据文本转为向量（使用 text-embedding-3-small）
- 内存索引，支持余弦相似度语义检索
- 拼接标题 + 摘要 + 研究类型作为检索文本

#### Reranker（重排序器）
- LLM 交叉编码器精排（对 Top-15 候选逐条评分 0-10）
- **融合分数公式**：`0.4 × 向量相似度 + 0.4 × LLM评分 + 0.2 × 等级权重`
- 等级权重：指南 1.0 / 说明书 0.9 / 英文论文 0.8 / 中文论文 0.7 / 试验 0.6 / 会议 0.5

#### ContextInjector（上下文注入器）
- Top-K 证据组装为结构化上下文
- Token 预算控制（默认 3000 tokens）
- 低相关性证据自动过滤（阈值 0.3）

### 12.4 数据流

```python
# 完整 RAG 调用示例
rag = RAGPipeline(knows_client, llm_gateway)

# 返回: (注入上下文文本, 全部证据列表, 检索统计)
context, evidences, stats = await rag.retrieve_and_inject(summary)
# stats = {"queries": 6, "retrieved": 35, "reranked": 8, "injected": 5}
```

## 13. 多 Agent 协作：随访计划生成

### 13.1 四个 Agent 角色

| Agent | 角色 | 输入 | 输出 |
|-------|------|------|------|
| Triage Agent | 分诊官 | 出院小结 | PICO框架 + 检索计划 + 风险标记(HIGH/MEDIUM/LOW) |
| Search Agent | 检索员 | 检索计划 | 排序后的循证证据 + 相关性评分 |
| Planner Agent | 规划师 | 患者数据 + 证据 | 时间线随访计划 |
| Review Agent | 审核员 | 计划 + 原始数据 | 6维度评分 + 审核意见 |

### 13.2 审核员 6 维度评分

```
1. 安全性 (SAFETY)    — 高风险用药/危险症状/特殊患者
2. 循证性 (EVIDENCE)  — 证据支撑/质量/局限性标注
3. 完整性 (COMPLETENESS) — 时间线/药物/复查覆盖
4. 可执行性 (ACTIONABILITY) — 具体可操作/目标值
5. 一致性 (CONSISTENCY) — 无矛盾/饮食-药物无冲突
6. 合规性 (COMPLIANCE) — 免责声明/无剂量调整/遵医嘱
```

**通过规则**：
- 全维度 ≥ 7 → approve
- 有维度 5-6 → minor（小修后自动通过）
- 有维度 < 5 或 safety < 7 → major（回流重修，最多 2 轮）

### 13.3 编排流程

```
Triage → Search(RAG) → Planner → Review
                                    │
                          ┌─────────┼─────────┐
                          ▼         ▼         ▼
                      approve    minor     major
                        │         │         │
                      输出计划   小修通过   回流Planner
                                            (最多2轮)
```

### 14.2 意图分类

| 意图 | 关键词触发 | 后续路径 |
|------|-----------|---------|
| report（汇报指标） | 血糖/血压/体温/测了 | 提取数值 → 趋势分析 → 预警 → 回复 |

每次 AI 生成回复后，经过 Guardrails Pipeline 审核（见第 18 节），不通过则重新生成（最多 1 次）。

## 15. 趋势分析预警系统

### 15.1 核心创新：不只看值，还看趋势

```
时间轴：  Day1    Day3    Day7    Day14   Day30
血糖：    8.2 →   7.8 →   7.1 →   6.8 →   6.5  ✅ 持续改善
血压：    145  →  150  →  155  →  160  →  165   🔴 持续恶化
体重：    70   →  69.5 →  70   →  69   →  70    🟢 稳定
```

### 15.2 趋势方向判定

| 方向 | 判定逻辑 | 含义 |
|------|---------|------|
| IMPROVING | 均值下降且指标"低=好"（如血糖）| 持续改善 |
| STABLE | 变化幅度 < 0.1 | 稳定 |
| WORSENING | 均值上升且指标"低=好" | 持续恶化 |
| FLUCTUATING | 正负交替 > 60% | 波动大 |
| INSUFFICIENT | 数据点 < 2 | 数据不足 |

### 15.3 预警升级规则（V2）

| 规则 | 条件 | 结果 |
|------|------|------|
| R1 | 当前值超红色阈值 | → RED |
| R2 | 连续3次恶化 + 超黄色阈值 | → RED（趋势升级）|
| R3 | 当前值超黄色阈值 | → YELLOW |
| R4 | 连续2次恶化（即使在范围内）| → YELLOW |
| R5 | 预测下次值将超阈值 | → YELLOW（预防性）|

### 15.4 线性外推预测

```python
# 简单线性外推：预测下次指标值
prediction = current_value + change_rate  # change_rate = 最近3次差值均值
# 如果 prediction 超出目标范围 → 触发预防性预警
```

## 16. 患者画像引擎

### 16.1 交叉分析维度

```python
PatientProfileEngine.analyze(summary) → RiskProfile

RiskProfile:
  overall_risk_level: "low" | "medium" | "high" | "critical"
  risk_factors: [...]           # 识别出的风险因素
  personalized_queries: [...]   # 个性化检索查询
  special_attention: [...]      # 特殊注意事项
  medication_interactions: [...] # 潜在药物相互作用
```

### 16.2 分析规则

| 维度 | 条件 | 影响 |
|------|------|------|
| 年龄 ≥ 65 | 老年患者 | 增加跌倒预防查询、药物代谢注意 |
| 年龄 ≥ 80 | 超高龄 | 多重用药审查、认知评估 |
| 用药 ≥ 5 种 | 多重用药 | 药物相互作用高风险 |
| 用药 ≥ 3 种 | 多药联用 | 关注相互作用 |
| 高风险药物组合 | 华法林+阿司匹林等 | 特定风险标记 |
| 共病 | 糖尿病+高血压 | 心血管事件风险增加 |
| 指标异常 | 血糖>10 / 血压>160 | 控制不佳标记 |
| 手术史 | 造影/手术 | 术后特殊关注 |

### 16.3 对检索的影响

患者画像的 `personalized_queries` 会注入 RAG Pipeline 的查询构造器：
- 老年患者 → 额外查"老年患者特殊注意事项"
- 多重用药 → 额外查"多药联用 相互作用"
- 共病 → 额外查"共病管理指南"

## 17. 证据分级体系

### 17.1 证据金字塔

```
         ▲  A级: 临床指南 (最高权威)
        ▲▲▲  B级: RCT / 系统综述 / Meta分析
       ▲▲▲▲▲  C级: 观察性研究 / 队列研究 / 病例对照
      ▲▲▲▲▲▲▲  D级: 会议摘要 / 在研试验
     ▲▲▲▲▲▲▲▲▲  E级: 药品说明书 / 专家意见
```

### 17.2 自动分级逻辑

1. **基础等级**：根据数据源类型给默认等级
2. **研究类型调整**：如 paper 源但 study_type 是 RCT → 升级到 B
3. **Meta 分析检查**：检查是否包含异质性分析（I²）、效应量+CI
4. **置信度计算**：`基础 0.7 + 影响因子加分 + 摘要长度加分 + 年份加分 - Meta缺陷扣分`

### 17.3 低证据标注

C/D 级证据在前端展示时自动标注：
```
[C级证据 - 观察性研究，可能存在偏倚，仅供参考]
[D级证据 - 会议摘要，结论未经同行评审]
```

## 18. Guardrails 安全护栏

### 18.1 四层护栏架构

```
AI 原始回复
    │
    ├──▶ Layer 1: 药物安全护栏
    │       禁止：建议停用/调整剂量/换药
    │       处理：自动替换为"请咨询医生"
    │
    ├──▶ Layer 2: 诊断安全护栏
    │       禁止：确诊性表述（"您患了"/"确诊为"）
    │       处理：自动替换为"建议进一步检查"
    │
    ├──▶ Layer 3: 矛盾检测护栏（LLM 辅助）
    │       检测：回复是否与循证证据矛盾
    │       处理：标记重新生成
    │
    ├──▶ Layer 4: 合规护栏
    │       确保：包含免责声明（仅供参考/遵医嘱/就医）
    │       处理：自动追加免责声明
    │
    └──▶ 通过 → 输出给用户
```

### 18.2 药物安全关键词

| 禁止表述 | 自动替换为 |
|---------|-----------|
| 建议您停用 | 建议您咨询医生是否可以停用 |
| 可以停止服用 | 请与医生讨论是否调整用药 |
| 减少剂量 | 请勿自行调整剂量，遵医嘱服药 |
| 增加剂量 | 请勿自行调整剂量，遵医嘱服药 |
| 不需要吃药 | 请遵医嘱按时服药 |
| 换用 | 如需换药请咨询医生 |

### 18.3 矛盾检测（Layer 3）

使用轻量 LLM 调用检查 AI 回复是否与注入的循证证据矛盾：
- 输入：AI 回复 + 循证证据上下文
- 输出：`{"has_contradiction": bool, "details": str}`
- 如矛盾 → 标记 `REGENERATE`，触发重新生成

## 22. 技术亮点总结（评委演示用）

| 序号 | 亮点 | 说明 | 对比竞品 |
|------|------|------|---------|
| 1 | RAG Pipeline | 检索→向量化→重排序→注入，非简单 API 调用 | 大多数项目只调 API |
| 2 | Multi-Agent | 4个Agent协作+审核循环，非单次 LLM 调用 | 大多数项目单次调用 |
| 3 | LangGraph 状态机 | 多状态流转+条件路由+工具调用 | 大多数项目简单对话 |
| 4 | 趋势预警 | 连续指标趋势分析+预防性预警 | 大多数项目只看单次值 |
| 5 | 患者画像 | 年龄/共病/多药交叉分析→个性化 | 大多数项目无个性化 |
| 6 | 证据分级 | 自动A-E分级+低证据标注+置信度 | 大多数项目不分级 |
| 7 | 安全护栏 | 4层护栏+矛盾检测+自动修正 | 大多数项目只有免责声明 |
